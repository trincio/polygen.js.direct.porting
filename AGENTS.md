# AGENTS.md — Polygen.js Direct Porting

## Project Overview

This is a **direct, faithful translation** of the original OCaml Polygen text generator
(Alvise Spano', 2002) into a single vanilla JavaScript file (`polygen.js`), browser-compatible,
zero dependencies.

The OCaml source is at `../Polygen/src/` (sibling directory, not part of this repo).

**Repo:** https://github.com/trincio/polygen.js.direct.porting

- `node test.js` → **67/67 passed** (unit tests, mulberry32 PRNG)
- `node tests/test.golden.js` → **74/99 passed** (OCaml golden comparison, `prng:'ocaml'`)

### What works
All synthetic grammars pass golden comparison: alternatives, concat, capitalize, optional,
non-terminals, labels, multi-label selectors, assignment `:=`, mobile groups `{}`, unfold `>`,
deep unfold `>><<`, lock `<`, weighted `++/--`, repetition `()+`, local declarations,
epsilon, complex combinations.

### What fails (golden comparison)
25 failures in real-world grammars. Two categories:

**Category 1: Trailing newline** (~6 cases: haiku, trial)
Our output has a trailing `\n` that OCaml's output doesn't. This is likely in `post()` or
in how we handle `\n` as a terminal. Quick fix expected.

**Category 2: Content divergence** (~19 cases: antani, blues, zen, bofh, unieuro, genio, manager, girotondi)
Completely different output with the same seed. These indicate real bugs in the JS porting —
either in parsing, preprocessing, or generation of complex grammar constructs.

## Architecture

`polygen.js` is a single IIFE exposing `window.Polygen` (browser) or `module.exports` (Node).

### Pipeline
```
PRNG (mulberry32 default, or OCaml-compatible LFG)
  ↓
Lexer        (tokenize + stripComments)     → token array
  ↓
Parser       (parse)                         → Absyn0 (AST)
  ↓
Preprocessor (preprocess)                    → Absyn1 (processed AST)
  ↓
Generator    (generate)                      → output string
```

### Key Design Decisions

1. **Sentinel objects** for special terminals (`T_EPS`, `T_CAT`, `T_CAP`) — identity comparison
   with `===`, immune to collision with literal grammar strings.

2. **Env as associative list** — `pairs.concat(env)`, lookup scans left-to-right (OCaml `List.assoc`).

3. **Former/Latter tags** — `{tag:"F"|"L", val:x}` for mobile vs fixed atoms in permutation.

4. **Lazy env** — `envRef = {v: null}` populated after closure creation, for mutual recursion.

5. **Two PRNGs:**
   - `makeMulberry32(seed)` — default, fast, JS-idiomatic
   - `makeOcamlRandom(seed)` — exact port of OCaml 4.x `Random` module (MD5-seeded Lagged
     Fibonacci F(55,24,+) with bit-mixing). Used only for golden comparison (`opts.prng: 'ocaml'`).

6. **`permute()` matches OCaml order** — uses the same `expl`/`perm` recursive structure as `pre.ml`.

## How to Work

### Running tests
```bash
node test.js                # unit tests (67 tests, mulberry32 PRNG)
node tests/test.golden.js   # golden comparison (99 tests, OCaml PRNG)
```

### Debugging
- Parse issues: `console.log(tokenize(src))` before `parse()`
- Preprocess issues: `console.log(JSON.stringify(preprocess(parse(src)), null, 2))`
- Generation issues: inspect the compiled grammar object

### OCaml reference
The original OCaml sources are at `../Polygen/src/`. Key files:
- `prelude.ml` — utilities, LabelSet, permute
- `absyn.ml` — AST types (Absyn0, Absyn1, Absyn2)
- `lexer.mll` — lexer
- `parser.mly` — parser
- `pre.ml` — preprocessor (Absyn0 → Absyn1)
- `gen.ml` — generator (Absyn1 → string)

### Golden test infrastructure
- `tests/grammars/*.grm` — grammar files (synthetic + real from Polygen collection)
- `tests/cases.json` — test case definitions (grammar × seed)
- `tests/golden.json` — expected outputs from the real OCaml binary (auto-generated by GitHub Actions)
- `tests/test.golden.js` — comparison runner
- `.github/workflows/golden.yml` — builds OCaml Polygen on Ubuntu, generates golden.json

## Your Task: Fix the 25 Golden Test Failures

### Priority order

**Step 1 — Fix trailing newlines (quick win)**
Cases: `real-haiku/*`, `real-trial/*`
The output has a trailing `\n` the OCaml binary doesn't produce. Investigate how `\n` is
emitted — likely in `post()` or in how the grammar uses `\n` as a terminal.
The OCaml binary outputs text followed by its `-eof` string (default `\n`), and the golden
runner does `.trimEnd()` on OCaml output. Our JS might be including grammar-internal `\n`
differently. Compare how the lexer tokenizes `\n` in quoted strings.

**Step 2 — Investigate content divergences**
Start with the **smallest divergences** (e.g., `real-zen/42` where only numbers differ:
"quattro" vs "tre"). This suggests a difference in how many PRNG calls are made
before reaching the choice point — likely a subtle difference in preprocessing or
generation order.

Then move to larger divergences (blues, antani, genio etc.).

### Debugging strategy
For each failing case:
1. Read the `.grm` file to understand the grammar
2. Run our JS: `Polygen.generate(src, {seed: N, prng: 'ocaml'})`
3. Compare with `tests/golden.json` expected output
4. Add `console.log` at key points in the generator to trace PRNG calls
5. Cross-reference with the OCaml source (`gen.ml`, `pre.ml`) to find the divergence

### What NOT to do
- Don't change the PRNG implementation (it's verified correct on 74 cases)
- Don't modify `tests/golden.json` (it's the ground truth from the OCaml binary)
- Don't break existing passing tests (run `node test.js` after every change)
- Don't add dependencies
- Don't restructure `polygen.js` into multiple files

## API Reference

```js
// All-in-one
Polygen.generate(source, {
  start:    "S",           // start symbol
  labels:   ["sg", "m"],   // active labels
  seed:     42,            // PRNG seed (deterministic)
  prng:     'ocaml',       // 'mulberry32' (default) | 'ocaml'
  maxDepth: 200            // recursion limit
});

// Compile once, generate many
var grammar = Polygen.compile(source);
Polygen.generate(null, { grammar: grammar, seed: 42 });

// Static analysis
var report = Polygen.check(source); // {errors: [], warnings: []}
```
